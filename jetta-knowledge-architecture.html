Jetta — Knowledge & Learning Architecture

Recommendation for a dynamic-learning knowledge system · researched June 2026

The recommendation in one paragraph

Treat these as two separate layers. Dynamic learning (how the KB grows from resolved tickets) is the higher-value piece and you can ship it now without vectors — it's a human-gated Knowledge Loop, not model retraining. RAG (vector + hybrid + reranking) is a clean retrieval upgrade you adopt once the KB is actually growing or to fix semantic search. Do dynamic learning first; add RAG when the KB earns it. Never fine-tune the model, and never auto-publish to the KB.

The two layers (don't conflate them)

RETRIEVAL

RAG

How Jetta finds the right knowledge for a query. Vectors + hybrid search + reranking.

Solves: search quality at scale, semantic matching ("my mappings disappear" → the Save-before-close article).

GROWTH

Dynamic learning

How the knowledge improves over time from resolved tickets and dev fixes.

Solves: Jetta getting smarter; closing the gaps that caused yesterday's escalations.

You need both eventually. But they solve different problems and have very different effort/risk profiles. The growth layer is what makes Jetta feel like it's "learning" — and it's mostly plumbing you partly have, not ML.

Dynamic learning — what it actually is (and isn't)

Not fine-tuning / RLHF / weight updates. That's expensive, slow, hard to control, governance-heavy, and risks regressions. For a support agent in 2026 the industry consensus is clear: "learning" = a continuously curated knowledge base + a feedback loop, not retraining the model.

The closed loop (this is exactly the spec's §5 Knowledge Loop, generalized):

┌─ gap detected ──────────────────────────────────────────────┐ │ Jetta can't resolve → escalates to a human (dev/agent) │ │ │ │ │ human resolves it ◄────┘ ← the gold signal │ │ │ │ │ Jetta DRAFTS a KB article from the thread │ │ │ │ │ ★ HUMAN APPROVAL GATE ★ (Content Mgr edits/approves) │ │ │ ← never auto-publish │ │ approved → chunk → embed → upsert to KB (searchable now) │ │ │ │ │ next similar ticket → Jetta resolves it herself ───────────┘ └──────────────────────────────────────────────────────────────┘

Plus three supporting mechanisms:

  • Feedback signals → gap report. Log every Jetta run's outcome (resolved-no-escalation = good; reopened = wrong/gap; escalated = gap). Aggregate the escalation/reopen themes → that's your prioritized "what to document next" list.
  • Confidence-gated escalation. Low retrieval confidence/grounding → escalate instead of guessing (Jetta already does "don't invent → escalate"). Tune the threshold from live data.
  • Decay & conflict management. Re-ingest on product changes (changelog-driven), flag duplicates/conflicts, expire stale articles.
The one non-negotiable guardrail: a human approval gate before anything enters the KB. Auto-publishing agent-written articles is how a "learning" system silently learns wrong answers and amplifies them. The spec already mandates draft-only in v1 — keep it.

RAG — the retrieval upgrade (when the KB earns it)

Today Jetta does keyword/token-overlap search over 65 short articles. For 65 articles that's genuinely fine. RAG pays off when (a) the KB grows to hundreds+ of chunks via the loop above, or (b) you want semantic matching that keyword search misses. Adopt it as a clean swap of the search_knowledge_base implementation — the tool interface stays identical, so nothing else changes.

If internal/proprietary content goes into the KB, choose providers consciously (Claude+Voyage with enterprise terms, or fully self-hosted) — and keep the rule that internal content informs escalations, never customer replies.

Recommended rollout for Jetta

0

Feedback capture now · ~free

Log every Jetta run's outcome (tools used, resolved vs escalated, did the customer reply/reopen) to the store. Costs nothing, and it's the data foundation every "learning" feature needs. Start collecting immediately.

1

The Knowledge Loop highest ROI

Build spec §5: dev resolves an escalation in Slack → Jetta drafts a KB article → posted for human approval → on approval it's appended to the KB. This is the dynamic learning you asked for. Human-gated, and it needs no vector infra to start.

2

RAG upgrade when KB grows / to fix semantic search

Swap search_knowledge_base to Upstash Vector + AI-SDK embeddings + hybrid + rerank. Ingest the 65 articles + Freshdesk Solutions + approved loop articles. Interface unchanged, so it's an isolated change.

3

Gap analytics ongoing

Add to the Ops Console: deflection rate, escalation themes, reopened tickets, KB-gap report. This is the dial that tells you what to document next — the loop's steering wheel.

My honest take: start with Phase 0 + Phase 1. That delivers the "dynamic learning" you're after, is the spec's intent, is low-risk, and reuses the Slack/escalation plumbing already built. Add RAG (Phase 2) as a clean retrieval swap once the KB is genuinely growing — don't stand up vector infrastructure before there's enough knowledge to justify it.
