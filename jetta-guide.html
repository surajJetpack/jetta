🤖 Jetta

Autonomous AI support agent for Jetpack Apps & GetSign — how it works, how it's built, and how to use it.
Last updated June 2026.

In one line: Jetta reads a support ticket, decides what to do using a set of tools (search the knowledge base, look up the dev board, reply, escalate to Slack, file a bug on monday.com), and acts — autonomously, with humans in the loop for anything risky. She runs on Google Gemini 2.5 Pro, deployed on Vercel, and you drive/observe her through the Ops Console.

1What Jetta is

Jetta is an autonomous support agent for two products — Jetpack Apps (monday.com marketplace apps) and GetSign (e-signature for monday.com). She handles the support lifecycle: understands the ticket, answers from the knowledge base, asks for missing details, files bugs, escalates to the dev team, and follows up — escalating to humans when judgment is needed.

What she does

  • Answers technical & how-to questions, grounded in real product docs
  • Files bugs to the monday.com Dev board and escalates to Slack with full context
  • Posts replies & internal notes on Freshdesk tickets
  • Follows up after 24h and closes resolved tickets
  • Learns: dev resolutions become new KB articles (human-approved)

What she doesn't do (by design)

  • No proactive outreach / sales / upsell
  • No customer-facing claims about internal tooling (the monday Dev board is internal — never shared with customers)
  • No irreversible action without a human gate (cancellations, going live on a ticket)
  • English only at launch

2How she works — the request lifecycle

Jetta is a stateless webhook handler. Every interaction is one pass through this loop:

ticket event ──▶ POST /api/webhook ──▶ [verify secret] ──▶ [dedupe] (Redis) │ ┌───────────────────────────────────┘ ▼ build context: fetch the ticket + conversation, the customer's account, related monday Dev items │ ▼ ┌──── agent loop (Gemini + tools, up to 10 steps) ───────────┐ │ model picks a tool → handler runs it → result back to model │ │ search_knowledge_base · search_dev_board · reply_to_ticket │ │ create_dev_item · send_escalation · add_private_note · … │ └──────────────────────────────────────────────────────────────┘ │ ▼ actions executed (or simulated, in dry-run) → reply posted │ ▼ log the run + outcome (Redis) ; if resolved → schedule 24h follow-up

The model isn't told "do step 1, 2, 3." The behaviour emerges from the system prompt's rules plus which tools it chooses. Workflows aren't hard-coded branches.

5Support workflows

Technical issue

Search the KB first → if an article resolves it, answer with the steps + link and ask the user to confirm → if not, ask targeted diagnostics → on a confirmed bug, check the Dev board: existing item → +1 and tell the user it's tracked; no item → create one + escalate to Slack. The internal monday link goes in the private note, never to the customer.

Escalation

Escalates to #jetta-escalations when: no KB answer after two turns, the user asks for a human, or it needs account-level debugging she can't do. The Slack message always carries the ticket URL, account URL, a summary, and what she already tried.

Knowledge Loop (how she learns)

escalation → dev resolves it in the Slack thread → @Jetta draft kb (Jetta drafts a KB article from the thread) → human reviews (Slack, or the /kb Approval queue) → @Jetta publish kb (approved → managed KB + vector index) → next similar ticket resolves itself

Human approval is mandatory — Jetta never auto-publishes to the KB.

24-hour follow-up

After sending a fix, Jetta logs a "resolution_sent" note → a job is stored in Redis. A daily cron checks it: customer replied → Jetta handles the reply; no reply → she sends a follow-up and closes the ticket.

Billing under rework

The spec assumed FastSpring billing, but GetSign actually bills through the monday.com Marketplace. The billing/churn flow is being reworked to reflect that; for now FastSpring tools are stubbed and a managed KB article tells Jetta to direct billing/refund questions to monday.com.

6Knowledge base & RAG

Jetta's answers are grounded in a knowledge base, not invented. Two sources:

SourceWhatEditable
Curated GetSign KB~66 articles distilled from getsign.io (features, how-tos, FAQ, a consolidated "supported workflows" reference)in git
Managed KBHuman-curated articles + approved Knowledge-Loop articlesin the /kb UI

Retrieval: the query is embedded (Gemini) and matched against the Upstash Vector index; the top articles are handed to the model. Managed + curated articles all live in the index. (Freshdesk Solutions articles were intentionally removed — quality wasn't trusted.)

Grounding rules (why she doesn't make things up)

  • Product-specific steps must come from a retrieved article — not from the model's general knowledge.
  • An article merely mentioning a feature isn't enough; it must actually contain the procedure.
  • If nothing covers it, she says she's confirming the steps / asks / escalates — she does not guess or cite a loosely-related article.

7Using the Ops Console

The console is an internal, admin-gated web app with three tabs. Access it with the admin key in the URL: https://<app>.vercel.app/?key=YOUR_ADMIN_SECRET (the key lives in Vercel env / with the team lead — treat it like a password).

Console tab — run & test tickets

  • System status — model + which integrations are live/stubbed.
  • Ticket runner — type a Freshdesk ticket ID and Run.
    • Dry-run ON (default): runs the full pipeline and shows the trace + the reply she would send — but posts nothing. Safe on any ticket.
    • Dry-run OFF: she acts for real — but only on allowlisted tickets (others are auto-forced to dry-run).

Knowledge Base tab (/kb)

  • Test retrieval — type a query, see exactly which articles + scores Jetta would retrieve.
  • Approval queue — review & Approve/Reject Knowledge-Loop drafts.
  • Managed articles — add / edit / delete; every change auto-syncs to the vector index.
  • Curated articles — browse-only (edited in git).

Insights tab (/analytics)

  • Gap analytics — runs logged, deflection rate, escalated/reopened counts, "document these next" gaps, recurring themes.
  • Activity log — every run with its full tool trace, KB hits + scores, the reply, token usage, timing, and errors.

Slack admin commands

In the designated channel (admin users only): @Jetta status ticket #123, @Jetta open tickets, @Jetta extend trial for x@y.com 7 days, @Jetta apply discount CODE to x@y.com, @Jetta cancel account x@y.com (needs a 2nd admin to confirm), and in an escalation thread @Jetta draft kb / publish kb. (Requires Slack Event Subscriptions wired — see gaps.)

11Current status & known gaps

Live & working: Freshdesk + monday + Slack(outbound) integrations · Gemini reasoning · RAG vector KB you manage from /kb · feedback analytics + detailed run logs · Knowledge Loop · ticket allowlist · admin security · deployed on Vercel.

Known gaps / open items
  • Billing flow — FastSpring is stubbed; the churn/cancel tools don't reflect monday.com Marketplace billing yet.
  • Freshdesk auto-trigger — not wired; tickets are run manually via the console until a Freshdesk automation POSTs to the webhook.
  • Slack @Jetta commands — handler is built but needs Slack Event Subscriptions + channels:history scope to fire.
  • monday trial extension / platform discount — need the GetSign Marketplace app's developer credentials.
  • Non-GetSign product KB (e.g. Vlookup Auto-Link) — coverage was removed with Freshdesk; restore by crawling those product sites.
  • Escalation wording — occasionally over-promises ("a fix will be deployed"); minor prompt tuning.
  • Reranker / grounding-verify — deferred until logs show it's needed.

12Glossary / FAQ

Dry-run — run the full pipeline but simulate (don't execute) any write.
Allowlist — the set of ticket IDs Jetta may act on for real.
Managed article — a KB article curated at runtime (UI or Knowledge Loop), stored in Redis + the vector index.
Knowledge Loop — turning a dev's resolution into a human-approved KB article.
Grounding — basing answers on retrieved KB content rather than the model's own guesses.
RAG — Retrieval-Augmented Generation: search the KB, feed the hits to the model.

Q: Will Jetta touch a real customer ticket I didn't intend? No — there's no auto-trigger yet, and live writes are gated by the allowlist. She only acts on tickets you run (and only writes on allowlisted ones).

Q: How do I see what she did? Insights → Activity Log: every run with its trace, the KB it used, and the reply.

Q: How do I fix a wrong answer? Add/edit a managed article in /kb — it's embedded immediately and Jetta will use it on the next matching ticket.
